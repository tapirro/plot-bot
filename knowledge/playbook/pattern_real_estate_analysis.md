# Pattern: Real Estate Deal Analysis

> Предложение → Intake → Research → 6-осевой анализ → VERDICT + QUESTIONS

**Происхождение:** Кристаллизовано 2026-03-05 из анализа Olympus Residence и Кведа Самеба.
**Аналог:** `pattern_meeting_insights.md` (анализ встреч), но для инвестиционных предложений по недвижимости.

---

## problem: Problem

Предложения по недвижимости приходят хаотично — чат, ссылка, PDF, фото — и требуют
быстрой оценки. Без структурированного подхода анализ субъективен: акцент на одном
аспекте (цена) при игнорировании других (ликвидность, юридические риски, достоверность
источника). Грузинская специфика (кадастровая система, налоги, ограничения для иностранцев)
добавляет слои неочевидных рисков, которые легко упустить при ad-hoc оценке.

## solution: Solution

Структурированный pipeline из 5 этапов (INTAKE → RESEARCH → ANALYSIS → DELIVERABLES →
PRESENT), выполняемый за 10-15 минут. Каждое предложение оценивается по 6 осям:
VALUE, LOCATION, RISK, CREDIBILITY, LIQUIDITY, URGENCY — с числовыми оценками и
обоснованиями. Claims продавца классифицируются по уровню достоверности (C1-C3).
Research проводится параллельно субагентами (market + legal + documents).
Кадастровые данные проверяются через NAPR API и ArcGIS REST напрямую (не через браузер).
Результат — два документа: VERDICT.md (6-осевой анализ с вердиктом) и QUESTIONS.md
(вопросы для продавца, повышающие confidence при ответе). Кейс-папка сохраняет
полный audit trail для будущих итераций.

## Когда использовать

- Вадим получает предложение по недвижимости (ссылка, файл, чат)
- Нужна структурированная оценка: стоит ли входить, какие вопросы задать
- Любой тип: земля, квартиры, коммерция, девелопмент, фонд

## Входные данные

- Материалы от продавца (чат, PDF, фото, Google Sheets)
- Контекст участников (`knowledge/entities.md`)
- Грузинская специфика (секция ниже)

## Типы сделок

Тип определяется на входе и влияет на акценты анализа:

| Тип | Код | Специфика анализа |
|-----|-----|-------------------|
| Земля | `LAND` | Кадастр, назначение, разрешение на строительство, коммуникации |
| Квартира | `APARTMENT` | Застройщик, стадия строительства, аналоги продаж, УК |
| Дом | `HOUSE` | Техническое состояние, земля, коммуникации, соседство |
| Коммерция | `COMMERCIAL` | Арендный поток, якорные арендаторы, vacancy rate |
| Девелопмент | `DEVELOPMENT` | Разрешения, бюджет строительства, timeline, exit strategy |
| Фонд | `FUND` | Структура фонда, track record управляющего, комиссии, lock-up |

## 6 осей анализа

| # | Маркер | Ось | Что оцениваем | Шкала |
|---|--------|-----|---------------|-------|
| 1 | `VALUE` | Ценность | Дисконт к рынку, доходность, потенциал роста | 1-5 |
| 2 | `LOCATION` | Локация | Район, инфраструктура, транспорт, экология, перспективы | 1-5 |
| 3 | `RISK` | Риски | Юридические, строительные, рыночные, политические | 1-5 (инвертирована: 5 = низкий риск) |
| 4 | `CREDIBILITY` | Достоверность | Источник предложения, документы, track record продавца | 1-5 |
| 5 | `LIQUIDITY` | Ликвидность | Скорость выхода, rental demand, resale market | 1-5 |
| 6 | `URGENCY` | Срочность | Временные рамки, конкуренция покупателей, дедлайны | E1-E4 |

### Шкала срочности (E-уровни)

| Уровень | Описание | Таймфрейм |
|---------|----------|-----------|
| `E4` | Решение нужно сегодня | Часы |
| `E3` | Решение на этой неделе | Дни |
| `E2` | Можно подумать | Недели |
| `E1` | Нет давления | Месяцы |

## Двумерная классификация claims

### Confidence (уровень достоверности)

| Уровень | Описание | Пример |
|---------|----------|--------|
| `C3` | Верифицировано | Кадастровая выписка, закрытая сделка, госреестр |
| `C2` | Проверяемые данные | Таблица от продавца с конкретными цифрами, листинги |
| `C1` | Маркетинговые материалы | Рекламная презентация, устные заявления, "аналогов нет" |

### Матрица приоритетов

```
              Высокая ценность    Средняя ценность    Низкая ценность
C3            INVESTIGATE          CONSIDER             PASS
C2            DUE DILIGENCE        CONSIDER             PASS
C1            VERIFY FIRST         LOW PRIORITY          PASS
```

## Workflow выполнения

```
Вадим: "вот предложение" (ссылка / файл / чат)
         │
         ▼
[1] INTAKE (1-2 мин)
    ├── mkdir work/intake/deals/YYYY-MM-DD_slug/
    ├── Скопировать/скачать материалы → source/
    ├── Нормализовать чат → source/chat.md
    ├── Прочитать фото, PDF → первичное понимание
    └── Заполнить README.md (карточка кейса)
         │
         ▼
[2] RESEARCH (3-5 мин, параллельно)
    ├── Субагент 1: WebSearch рыночные цены + аналоги
    ├── Субагент 2: WebSearch/Fetch юридика + кадастр
    └── Субагент 3: Анализ фото + PDF документов
         │
         ▼
[3] ANALYSIS (3-5 мин)
    ├── Свести данные из research/
    ├── Заполнить 6 осей с обоснованиями
    ├── Сценарный анализ (если применимо)
    ├── Выявить красные флаги
    └── Сформулировать вопросы
         │
         ▼
[4] DELIVERABLES
    ├── analysis/VERDICT.md — главный документ
    ├── questions/QUESTIONS.md — для отправки продавцу
    └── Обновить README.md (axes, confidence, status)
         │
         ▼
[5] PRESENT
    └── Показать Вадиму: radar + вердикт + вопросы
```

**Target time:** 10-15 минут на полный анализ.

## Чеклист исследования

### Рыночные данные (WebSearch)

- `"Батуми недвижимость цены {тип} {год}"` — общий рынок
- `"Batumi real estate price per sqm {type} {year}"` — англоязычные источники
- `"{название проекта} отзывы"` — репутация застройщика
- `"Батуми {район} инфраструктура"` — локация
- `"Грузия налог на недвижимость иностранцы"` — налоги и право

### Кадастр и юридика (API — не WebFetch!)

**ВАЖНО:** maps.napr.gov.ge и maps.gov.ge — JavaScript SPA, WebFetch возвращает пустой HTML. Использовать API напрямую.

**NAPR API (регистрация собственности, ипотеки):**
```bash
python3 tools/scripts/napr_lookup.py 05.32.12.220.01.01.513
python3 tools/scripts/napr_lookup.py --file codes.txt  # пакетная проверка
```
- Endpoint: `POST https://naprweb.reestri.gov.ge/api/search`
- Без авторизации. Body: `{"cadcode": "XX.XX.XX.XXX.XX.XX.XXX"}`
- Возвращает: историю регистраций, стороны сделок, ипотеки, адрес
- Кадастровый код: `ZZ.SS.QQ.PPP.BB.TT.UUU` (зона.сектор.квартал.участок.здание.башня.единица)

**ArcGIS REST (карта, площади, этажность):**
```bash
python3 tools/scripts/napr_lookup.py --parcel 053212220
```
- Endpoint: `http://gisappsn.reestri.gov.ge/ArcGIS/rest/services/CadRepGeo/MapServer/`
- **Только HTTP** (не HTTPS — SSL cert для другого домена)
- Layer 14 = участки Аджарии, Layer 12 = здания
- UNIQ_CODE = 9 цифр без точек (05.32.12.220 → 053212220)

**Законодательство:**
- `https://matsne.gov.ge` — законодательство Грузии (WebFetch работает)

### Telegram-экспорт (intake)

```bash
python3 tools/scripts/telegram_export.py /path/to/ChatExport/ --summary
python3 tools/scripts/telegram_export.py /path/to/ChatExport/ --after 2026-03-06
python3 tools/scripts/telegram_export.py /path/to/ChatExport/ --photos --copy-to source/photos/
```

### Инструменты AWR

- `tools/scripts/napr_lookup.py` — кадастровая проверка (NAPR + ArcGIS)
- `tools/scripts/telegram_export.py` — парсинг Telegram-экспортов
- `WebSearch` — рыночные цены, отзывы, новости
- `WebFetch` — Google Sheets продавца, matsne.gov.ge, документы по ссылкам
- `Read` — PDF-документы, фото (мультимодальный анализ)
- Субагенты — параллельный market research + legal research + photo analysis

## Грузинская специфика (reference)

| Тема | Инструмент | Что проверять |
|------|------------|---------------|
| Кадастр | `napr_lookup.py` (NAPR API) | Номер, обременения, ипотеки, история переходов |
| Карта | `napr_lookup.py --parcel` (ArcGIS) | Площадь участка, этажность, площадь застройки |
| Градостроительство | matsne.gov.ge | Зонирование, коэффициенты застройки |
| Налоги | matsne.gov.ge | На имущество (до 1%), на доход |
| Иностранцы | matsne.gov.ge | С/х земля — запрет для физлиц; через ГК — можно |
| Регистрация | napr.gov.ge (офлайн) | 1-4 дня, ~$50-200, нотариальное заверение |

### Налоговый режим (summary, уточнено из практики R4)

- **Покупка:** нет налога на покупку (transfer tax = 0 в Грузии)
- **Владение:** налог на имущество до 1% от рыночной стоимости
- **Продажа (физлицо):** 5% от прибыли (де-факто часто не применяется, но норма есть)
- **Продажа >6 кв/год на одного человека:** налоговая может квалифицировать как бизнес → 20%
- **Схема:** оформление на 3-4 физлиц, ≤6 продаж/год на каждого
- **Продажа (через ГК):** 15% корпоративный налог
- **Аренда:** 5% для физлиц-нерезидентов в статусе "Virtual Zone" или 20% стандарт
- **Форма сделки:** прямая продажа через Минюст (не переуступка) — если есть кадастровые номера

> Всегда верифицировать через matsne.gov.ge — налоговые правила меняются.

## Оценка строительной динамики

При анализе объектов в стадии строительства:

### Фото-сравнение (парные ракурсы)

1. Запросить фото **текущее + N месяцев назад** (минимум 2 точки)
2. Снять **5 стандартных ракурсов:** фронт, боковой, с угла, ближний план нижних этажей, вид на обе башни
3. Оценить по каждому ракурсу:

| Элемент | Что смотреть |
|---------|-------------|
| Фасад | % закрытия (панели/штукатурка), наличие лесов |
| Окна | Установлены? Бренд? Защитная плёнка? |
| Кран | Активен? Где расположен? |
| Нижние этажи | Внутренние работы видны? |

### Экстраполяция timeline

Формула: `(% оставшейся работы) / (% работы за измеренный период) × период = оставшееся время`

Коррективы:
- **Параллельность:** если застройщик строит последовательно (как Olympus), вторая башня начнётся только после первой
- **Масштаб:** большее здание ≠ пропорционально больше времени (40 этажей = 40 этажей, но периметр/этаж больше)
- **Сезонность:** зима в Батуми = дождь, замедление внешних работ
- **Приоритет:** какая башня строится первой → там timeline реалистичен

### Влияние на оценку

| Прогресс фасада | RISK | LIQUIDITY | Рекомендация |
|----------------|------|-----------|-------------|
| >80% | 3.5+ | 4.0 (ипотека скоро) | INVESTIGATE |
| 40-80% | 3.0 | 3.0 (ипотека через полгода-год) | CONTINUE |
| <40% | 2.0-2.5 | 2.0-2.5 (только кэш) | Дисконт или отложить |
| 0% (каркас) | 1.5-2.0 | 1.5-2.0 | Сильный дисконт или PASS |

## Заметки по реализации

- **Батчинг субагентов:** 3 параллельно (market + legal + documents)
- **Контекст:** каждый субагент получает тип сделки и паттерн для фокуса
- **Идемпотентность:** если VERDICT.md уже существует — спросить перед перезаписью
- **Качество:** при C1-данных всегда помечать "требует верификации"
- **Приватность:** не публиковать финансовые детали без маркировки
- **Итерации:** после получения ответов на QUESTIONS.md — обновить VERDICT.md (повысить confidence)
